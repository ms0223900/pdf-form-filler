# User Stories / 使用者故事拆解

## Purpose / 目的

將需求拆解為個別小單元的實作任務，讓每個任務都能被獨立追蹤、實作與驗收。

使用本 skill 時，Agent 應扮演 Technical PM / System Analyst，從需求文件、模組拆解文件與解決方案中萃取可執行的 user stories。

## When to Use / 使用時機

啟用本 skill 的典型情境：

- 使用者要求「拆 user stories」、「拆使用者故事」、「拆實作任務」。
- 使用者提供需求文件、模組拆解文件、解決方案文件，並希望整理成可追蹤的任務。
- 使用者希望每個任務都有明確輸入輸出、驗收條件與依賴關係。
- 使用者要求將任務存成個別 markdown 文件。

## Required Inputs / 必要輸入

優先收集以下資料：

- **需求**：需求文件、PRD、ticket、規格或使用者描述。
- **模組拆解**：模組、功能區塊、資料流或依賴關係拆解。
- **解決方案**：技術方案、架構決策、API / UI / 資料模型設計。

若資訊不足，先用一個簡短問題請使用者補充最缺的資料；若使用者要求先產出草稿，可標記假設與待確認項目。

## Workflow / 操作流程

1. **理解脈絡**
   - 讀取需求、模組拆解與解決方案。
   - 摘要目標、受影響模組、主要角色、已知限制與依賴。

2. **拆成小單元任務**
   - 每個任務都應能獨立完成，不要過大。
   - 若任務跨越多個模組或需要多階段交付，繼續拆分。
   - 任務之間的依賴關係要明確，可參考模組拆解的依賴關係。

3. **補齊任務規格**
   - 每個任務必須包含：
     - **輸入格式**：需要什麼輸入資料、props、API payload、檔案或設定。
     - **輸出格式**：會產生什麼畫面、資料、檔案、API response 或狀態變化。
     - **驗收條件**：使用 checklist 表示，條件需可驗證。
     - **依賴關係**：列出前置任務、模組或外部條件。
     - **優先級**：使用 `P0`、`P1`、`P2`。
     - **相關功能**：對應功能模組、章節或需求編號。

4. **輸出個別文件**
   - 每個任務輸出為獨立 markdown 文件。
   - 文件名稱使用 `{任務名稱}.md`；若已有編號規則，沿用既有規則，例如 `US-002-查看我們的日常文章列表.md`。
   - 若實際寫檔，放在使用者指定的資料夾；未指定時先詢問目標位置。

5. **產出總覽 README（多 US 時）**
   - 若此批輸出有 **3 個或以上**的 US 文件，必須在**同一個目錄**額外產出一份 `README.md`。
   - 產出前先掃描目標目錄中**既有的 US 文件**，與新產出的合併計算，將新舊 US 一併納入 README.md。
   - `README.md` 內容結構：
     - **專案 / 功能簡述**：一句話說明這批 US 的目的。
     - **US 一覽表**：表格列出所有 US 編號、標題、優先級、依賴關係、簡要說明（新舊合併）。
     - **依賴關係圖**：以純文字箭頭圖（ASCII diagram）呈現 US 之間的依賴（包含新舊 US 的跨批次依賴）。
     - **依賴說明**：逐條解釋各 US 的前置任務與原因。
     - **建議開發順序**：將 US 分波次（Wave），說明哪些可平行、哪些需等待。
   - 若目標目錄已有 `README.md`，直接**覆蓋更新**（重新計算新舊 US 的完整關係）。
   - 若 US 數量為 **1~2 個**，不強制產出 README.md（但可選擇性產出）。
   - 可參考 `docs/init-project-features/user-stories/README.md` 的格式與風格。

## User Story File Template / 文件模板

```markdown
### US-XXX：{任務名稱}

**作為** {角色}
**我想要** {行為}
**以便** {價值 / 目的}

**輸入格式**：
- {此任務需要的輸入資料、API、props、設定或前置內容}

**輸出格式**：
- {此任務完成後產生的結果、畫面、資料或狀態變化}

**驗收條件**：
- [ ] {可驗證條件 1}
- [ ] {可驗證條件 2}
- [ ] {可驗證條件 3}

**依賴關係**：
- {前置任務或模組；若無則寫「無」}

**優先級**：P0 / P1 / P2
**相關功能**：{功能名稱或需求編號}
```

## Example / 範例

```markdown
### US-002：查看「我們的日常」文章列表

**作為** 求職者
**我想要** 瀏覽「我們的日常」文章列表
**以便** 了解公司內部日常活動與文化

**輸入格式**：
- 文章列表資料，包含標題、發布日期、標籤與詳情頁連結

**輸出格式**：
- 顯示可點擊的「我們的日常」文章列表區塊

**驗收條件**：
- [ ] 區塊標題顯示「我們的日常」與「Career News」
- [ ] 每篇文章顯示標題、發布日期、標籤徽章
- [ ] 每篇文章前有藍色箭頭圖示，表示可點擊
- [ ] 文章列表垂直排列，每篇文章間有適當間距
- [ ] 預設每頁顯示 3 篇文章
- [ ] 標題可點擊，連結至文章詳情頁
- [ ] 日期格式為 YYYY-MM-DD
- [ ] 響應式設計在各種裝置上正常運作

**依賴關係**：
- 文章資料來源已定義

**優先級**：P0
**相關功能**：文章列表區塊（2.1.2）
```

## Quality Checklist / 品質檢查

輸出前確認：

- [ ] 每個 user story 都符合「作為...我想要...以便...」格式。
- [ ] 每個任務都小到可以獨立實作與驗收。
- [ ] 每個任務都有輸入格式、輸出格式、驗收條件、依賴關係、優先級與相關功能。
- [ ] 驗收條件是可觀察、可測試的 checklist。
- [ ] 依賴關係沒有循環或含糊描述。
- [ ] 文件名稱清楚且方便後續追蹤。
- [ ] 若 US 數量 >= 3，已產出對應的 `README.md` 總覽文件。
